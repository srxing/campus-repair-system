## 一、项目概述

### 1.1 项目背景

校园后勤报修工作通常涉及宿舍、教学楼等多场景设备维护，如水电故障、网络异常、设施损坏等。传统报修方式存在流程分散、处理进度不透明、沟通成本高等问题。

因此设计一套基于移动端与Web管理端协同的校园智能报修系统，实现报修业务全流程数字化管理。

### 1.2 项目目标

本系统实现以下目标：

- 支持学生通过小程序快速提交报修
- 支持后台统一工单管理与派单
- 实现维修流程全生命周期跟踪
- 提供维修数据统计与可视化分析
- 实现多角色协同（学生/维修人员/管理员）
- 消息通知实时触达，提升沟通效率

### 1.3 系统范围

**覆盖范围：**

- 报修申请与提交
- 工单流转管理
- 维修派单与处理
- 评价反馈与回访
- 数据统计分析
- 消息通知与催办
- 系统管理与配置

**不包含：**

- 实体设备控制（IoT）
- 复杂第三方支付系统

## 二、系统架构设计

### 2.1 技术架构

| 端 | 技术选型 |
|----|----------|
| 学生端 | 微信小程序 / UniApp |
| 管理端 | Vue.js + Element UI |
| 后端 | Spring Boot + MyBatis-Plus |
| 数据库 | MySQL |
| 缓存 | Redis（可选） |
| 实时通信 | WebSocket（可选） |
| 数据可视化 | ECharts |
| 文件存储 | 本地文件存储（不依赖云OSS） |
| 地图服务 | 高德地图 API（路线导航） |

### 2.2 系统整体架构

```
学生小程序（报修端）
          ↓  HTTPS
     Spring Boot API
          ↓
   Web管理后台（Vue）
          ↓
        MySQL + Redis
```

### 2.3 架构说明

- 小程序负责数据采集与查询
- 后台负责流程控制与管理
- 后端提供统一 RESTful API 服务
- Redis 用于会话缓存与热点数据
- WebSocket 用于实时消息推送

### 5.1 登录与权限模块（统一）

系统支持多角色登录，并根据角色进入不同端。

  - 维修人员：工单处理权限
  - 管理员：全部权限

### 5.2 学生端（小程序）功能模块

#### 5.2.1 报修提交模块（核心）

**功能：**

- 选择报修类型：
  - 水电问题
  - 网络问题
  - 门窗损坏
  - 设备故障
  - 其他
- 填写问题描述（支持语音输入转文字）
- 选择报修位置（区域 → 楼栋 → 楼层 → 房间）
- 拍照上传（1~5张，本地文件存储）
- 提交工单

**业务规则：**

- 自动生成工单编号（规则见 7.1）
- 提交后状态：待分配
- 提交前展示常见问题自查（减少无效报修）

#### 5.2.2 工单查询模块

**功能：**

- 查看个人报修记录
- 查看工单状态
- 查看处理进度（含日志时间线）
- 按状态筛选（全部/待分配/已分配/维修中/已完成）

**状态展示：**

| 状态 | 说明 |
|------|------|
| WAIT | 待分配 |
| ASSIGNED | 已分配 |
| PROCESSING | 维修中 |
| DONE | 已完成 |
| CLOSED | 已关闭 |

#### 5.2.3 工单催办模块（拓展）

**功能：**

- 对超时未处理的工单进行催办
- 催办后通知管理员
- 每单限催办 3 次

**规则：**

- 待分配超过 24h 可催办
- 已分配超过 48h 可催办

#### 5.2.4 工单撤回模块（拓展）

**功能：**

- 学生撤回待分配工单
- 撤回后工单状态变为"已取消"

**规则：**

- 仅 WAIT 状态可撤回
- 已分派工单不可撤回

#### 5.2.5 评价模块

**功能：**

- 星级评分（1~5星）
- 文本评价
- 是否满意

**规则：**

- 仅已完成工单可评价
- 每单仅一次评价

## 六、核心业务流程设计

### 6.1 报修主流程

```
学生提交报修
      ↓
系统生成工单（WAIT）
      ↓
管理员分派维修人员（ASSIGNED）
      ↓
维修人员接单（PROCESSING）
      ↓
维修处理（填写说明+材料）
      ↓
完成维修（DONE）
      ↓
学生评价
      ↓
工单关闭（CLOSED）
```

### 6.2 工单状态流转

| 状态 | 说明 | 可执行操作 |
|------|------|-----------|
| WAIT | 待分配 | 分派、撤回、催办 |
| ASSIGNED | 已分配 | 接单、拒单、转单 |
| PROCESSING | 维修中 | 更新状态、上传结果 |
| DONE | 已完成 | 评价、回访 |
| CLOSED | 已关闭 | 归档、查看日志 |
| CANCELLED | 已取消 | 查看 |

**规则：**

- 状态单向流转（CANCELLED 除外）
- 不允许回退
- CLOSED 和 CANCELLED 为终态

### 6.3 催办流程

```
学生催办
      ↓
系统记录催办次数
      ↓
通知管理员
      ↓
管理员优先处理
```

## 九、系统数据模型（核心实体）

### 9.1 核心表结构

| 表名 | 说明 | 核心字段 |
|------|------|---------|
| `user` | 用户表 | id, username, password, phone, role, avatar, status |
| `repair_order` | 工单表 | id, order_no, type_id, student_id, worker_id, description, images, location, status, priority, create_time, finish_time |
| `repair_log` | 工单日志 | id, order_id, operator_id, action, from_status, to_status, remark, create_time |
| `repair_worker` | 维修人员 | id, user_id, group_id, skills, workload, status |
| `repair_type` | 报修类型 | id, name, description, group_id, sort_order |
| `rating` | 评价表 | id, order_id, stars, content, is_satisfied, create_time |
| `repair_material` | 维修材料（拓展） | id, order_id, name, quantity, price, create_time |
| `area` | 区域管理（拓展） | id, name, parent_id, level, sort_order |
| `notification` | 消息通知（拓展） | id, user_id, order_id, type, content, is_read, create_time |
| `knowledge` | 知识库（拓展） | id, type_id, title, content, view_count, create_time |

### 9.2 表关系

```
user 1 ──── N repair_order (student_id)
user 1 ──── N repair_order (worker_id)
repair_type 1 ──── N repair_order
repair_order 1 ──── N repair_log
repair_order 1 ──── 1 rating
repair_order 1 ──── N repair_material
area 1 ──── N repair_order
user 1 ──── N notification
repair_type 1 ──── N knowledge
```
